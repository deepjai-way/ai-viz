# 从架构图到万物可视化：一套 AI 驱动的通用方法论是怎么诞生的

---

## 第一章：起点——一次画图时的灵感

最近在做方案、画架构图的时候来了个灵感。

架构图、流程图、时序图、ER 图、部署图——每种图回答一个不同的问题。系统长什么样？请求怎么流转？数据怎么存？日常里大量的沟通，最终都收敛成一张图。但每次画图，真正花时间的不是"想清楚结构"，而是一堆隐性成本：选工具、学语法、调格式、对齐元素、导出图片。

想的部分花 20 分钟，画的部分花 2 小时。这个比例不对。

AI 编程工具已经能写几百行代码了——它能不能帮我画图？想到了就去试，然后就有了后面这些东西。

---

## 第二章：AI 能画图了，但问题不在 AI

答案是"能，但很粗糙"。

Claude Code /klɔːd koʊd/、Cursor /ˈkɜːrsər/、Windsurf /ˈwɪnd.sɜːrf/ 这些 AI 编程工具已经很强了。你跟它说"帮我画个微服务架构图"，它确实能生成一段 Mermaid 代码或者一个 Excalidraw /ɛks.kæl.ɪ.drɔː/ JSON。但用了几次之后，三个问题很明显：

**第一，没有方法论指导，质量不稳定。** 同一个需求问两次，出来的图结构完全不同。有时候布局合理，有时候节点挤成一团。AI 不知道什么是"好的架构图"。

**第二，每次从零描述需求，效率低。** 你得反复交代"分层画""连线要标注""配色别太花"。这些本该是默认规范，不该每次手动说。

**第三，没有统一标准，不同工具间不可复用。** 你在 Cursor 里攒了一套 prompt，换到 Claude Code 就得重写。

但这三个都是表面症状。用了一段时间之后，我意识到还有一个更根本的问题——**我们给 AI 的方式就错了。**

### 描述 vs 链接：两种完全不同的范式

传统方式是"你告诉 AI 画什么"。你得用自然语言描述："我要一个三层架构，上面是网关，中间是服务层有订单服务、用户服务、支付服务，下面是 MySQL 和 Redis。"

但等一下——这些信息是你发明的吗？不是。它们早就在你的设计文档里写着，在代码仓库的结构里躺着，在 DDL 里定义着，在 K8s YAML 里配着。你做的事情，本质上是**把已经存在的知识，人肉翻译成自然语言，喂给 AI**。

这就好比你面前放着一本英文书，你不让翻译官自己看，而是你一个字一个字念给他听，让他翻成中文。效率低是必然的，信息丢失也是必然的。

问题的根源不是 AI 画不好图——是我们给 AI 信息的方式不对。我们在"描述需求"，而不是"链接知识"。

### "翻译"这一步本不该由人来做

仔细想想这个过程：

1. 知识已经存在于某个载体中（设计文档、代码、DDL、API 规范……）
2. 你花时间阅读、理解、提炼出结构
3. 你把提炼后的结构用自然语言重新组织
4. AI 再从你的自然语言中提取结构
5. AI 按格式渲染成图

第 2、3 步完全是多余的。AI 本身就有能力从结构化/半结构化的源文件中直接提取结构。我们应该做的是：**让 AI 直接去读知识源，理解它的结构，然后画出来。**

这个认知转变是后面所有设计的起点。

核心洞察：**"AI 画图"和"AI 画好图"之间，隔着一套方法论。而方法论的第一性原理，是知识源。**

## 第三章：开源 Skill 生态的发现与改造

AI 编程工具都有一种扩展机制——Claude Code 叫 Skill，Cursor 叫 Rule，Copilot /ˈkoʊ.paɪ.lət/ 叫 Instructions。本质一样：把一段结构化指令注入 AI 的上下文，让它获得某个领域的专业能力。

我在 GitHub 上找到三个跟"AI 画图"直接相关的开源 Skill，逐个研究、改造、整合。

### 3.1 drawio-skill

**来源：** https://github.com/Agents365-ai/drawio-skill

这个项目的理念很直接：让 AI 直接输出 Draw.io 的 XML 文件。不走中间格式，不靠截图，直接生成可编辑的 `.drawio` 文件。

核心设计亮点：

- **Schema 约束**：定义了 `mxCell` 节点的所有合法属性（形状、配色、连线样式），AI 不会瞎编标签
- **布局规则**：规定了网格对齐、分层间距、复杂度分级策略
- **提示工程**：把"什么时候画什么图"的决策逻辑写进了 prompt

我在它基础上做的改造：提取出方法论内核（独立于格式）、增加中文指令支持、扩展支持的图种从 4 种到 14 种、增加 `design-language.yaml` 设计语言模板让配色项目级可配。

### 3.2 excalidraw-diagram-skill

**来源：** https://github.com/coleam00/excalidraw-diagram-skill

这个项目走的是另一条路：JSON Schema 驱动的白板图生成。Excalidraw 的文件格式是 JSON，每个元素都有精确的坐标、尺寸和样式属性。

核心设计亮点：

- **坐标系统**：左上角原点，X 右增 Y 下增，所有元素按网格对齐
- **元素类型定义**：rectangle、ellipse、diamond、arrow、text，每种有明确的语义用途
- **手绘风格控制**：`roughness: 1` 保证手绘感，`fontFamily: 5` 统一使用 Excalifont

我的改造：统一配色体系为语义槽位（primary/secondary/accent/muted），增加复杂度分级策略（节点数不同，间距策略不同），补充构图模式指引。

### 3.3 ian-xiaohei-illustrations

**来源：** https://github.com/helloianneo/ian-xiaohei-illustrations

这是转折点。

前两个 Skill 做的是"工程图"——架构图、时序图、流程图。而这个项目做的是"文章配图"。它让 AI 不只是画工程结构，而是把文章里的认知动作——一个判断、一个隐喻、一个状态变化——变成一张手绘图。

核心设计亮点：

- **小黑 IP**：一个黑色实心、白点眼、细腿的小角色，不是装饰，而是图中核心动作的承担者
- **8 种构图结构**：Workflow 流程、系统局部、前后对比、角色状态、概念隐喻、方法分层、地图路线、小漫画分镜
- **QA 检查清单**：生成后自检——小黑是不是只当装饰？画面是不是太满？像不像 PPT？
- **原创隐喻三步法**：抽象概念→物理动作，系统结构→低科技物件，让小黑承担动作

我的整合方式：把它作为一个插件接入路由系统。当内容类型是"科普文章/博客/概念解释"时，路由自动指向 ian-illustrator，而不是 drawio 或 mermaid。同时增加了"自动插入文章"能力——生成配图后，自动把图片引用插入到文章的正确位置。

## 第四章：顿悟——知识源才是第一性原理

改造完三个 Skill 之后，我发现它们有一个共同模式：

- **输入都是"知识源"（而不是用户描述）**：代码、设计文档、DDL、科普文章、接口规范
- **输出都是"图"（知识的可视化投影）**：架构图、流程图、时序图、手绘配图、概念图解
- **差别只在"路由"（AI 自动判断）**：根据知识源类型，选择最合适的可视化方式

DDL 进来出 ER 图，接口规范进来出时序图，科普文章进来出手绘配图。模式完全一样，只是映射关系不同。

但我没有在这里停下来。这个模式背后有一个更深的东西。

### 图的来源不是文章，是知识

一开始我以为这是个"画图工具"——你给一段文字，AI 画张图。但实际使用中发现，输入根本不是"一段文字"。输入是：一份 Swagger 文件、一组 K8s YAML、一段代码的 interface 声明、一篇设计评审文档、一个 PRD 的流程章节。

**这些东西有一个统一的名字：知识。**

知识不是我们为了画图而写出来的东西。知识是本来就存在的。它分散在你工作中使用的各种载体里：

- **代码仓库** — 放着系统的类结构、接口定义、调用关系
- **设计文档** — 放在 Confluence、Notion、飞书文档里，描述系统架构和设计决策
- **API 规范** — OpenAPI/Swagger 文件里定义了每个接口的输入输出
- **数据模型** — DDL 文件或 ORM 定义里躺着表结构和实体关系
- **运维配置** — K8s YAML、Docker Compose 定义了部署拓扑
- **需求文档** — PRD 里写着业务流程和状态机
- **架构决策记录** — ADR 里记录了为什么这样设计
- **会议纪要** — 里面藏着分歧、共识和决策
- **博客/文章** — 承载着概念解释和认知隐喻

**这些知识应该放在它该放的地方。** 代码就该在 Git 里，设计文档就该在 Confluence 里，DDL 就该在数据库里。ai-viz 不要求你把知识搬到别的地方，不创造新格式，不搞什么"架构描述文件"。

### ai-viz 的本质是"链接"

这是最关键的一步认知跃迁：

ai-viz 不是一个"画图工具"——它是一个**知识链接器**。它链接到不同的知识源，理解其结构，然后自动画出对应的图。

传统思路：用户描述需求 → AI 画图
ai-viz 思路：AI 读取知识源 → 理解结构 → 自动选择图种 → 渲染输出

区别在哪？

传统方式中，你是翻译官。你要先理解知识，再翻译给 AI。信息在翻译过程中必然丢失，而且每次都得重复这个劳动。

ai-viz 方式中，AI 直接是读者。它直接读原始知识源，从中提取结构。你省掉了"翻译"这一步，而且 AI 能看到的细节比你口述的更完整。

### 不同的知识源天然对应不同的图种

一旦你接受了"输入是知识源"这个前提，路由逻辑就变得自然而然了：

| 知识源类型 | 典型载体 | 天然对应的图种 | 回答的问题 |
|-----------|---------|--------------|-----------|
| 系统设计文档 | Confluence/Notion/Markdown | 架构图 | 系统长什么样？ |
| 接口规范/调用链 | OpenAPI/Swagger/代码 | 时序图 | 请求怎么流转？ |
| 业务规则/流程描述 | PRD/需求文档 | 流程图 | 业务怎么走？ |
| 数据模型/DDL | SQL/ORM 定义 | ER 图 | 数据怎么存？ |
| 代码结构 | 类/接口/模块定义 | 类图 | 对象怎么组织？ |
| 运维配置 | K8s YAML/Docker Compose | 部署图 | 怎么部署？ |
| 业务状态机 | 需求文档/状态定义 | 状态图 | 状态怎么变？ |
| 架构决策/数据互通 | ADR/设计文档 | 数据流图 | 数据从哪来到哪去？ |
| 科普内容/文章 | Markdown/博客 | 手绘配图 | 如何直观传达？ |
| 概念/认知/隐喻 | 自然语言描述 | 概念图解 | 如何可视化一个心智模型？ |
| 演进/历史叙述 | 规划文档/复盘 | 时间线图 | 事物怎么发展的？ |
| 并列对比分析 | 调研报告/选型文档 | 对比矩阵图 | 几个东西有什么区别？ |

你不需要告诉 AI "画个 ER 图"——你只需要把 DDL 文件交给它，它自己就知道该画 ER 图。这不是智能，这是逻辑必然。DDL 的结构决定了它只能被渲染成 ER 图，正如乐谱只能被演奏成音乐。

### 知识不需要被"翻译"

这是整个方法论的哲学内核：

**知识不需要被"翻译"给 AI——AI 应该直接去读知识源，理解它的结构，然后画出来。**

这意味着：
- AI 不是在"按你的描述画图"——它是在"从知识中提取可视化投影"
- 图表不是创作物——图表是知识的视图，就像数据库的 View 是数据的视图
- 同一份知识可以有多个投影——架构图是结构投影，时序图是行为投影，部署图是运维投影
- 选择画什么图，本质上是选择从哪个角度观察同一份知识

这是一个 **"知识 → 可视化"的哲学**，不是"描述需求 → 画图"的工具思维。

既然如此，就不应该做成"一个人的画图工具"，而应该做成一个通用框架。写技术文档的人能用，写科普文章的人也能用，做教程的人也能用。因为本质上大家都在做同一件事——把某种载体中的知识，转化为视觉表达。

## 第五章：方法论提炼——四层架构

我把这个通用模式抽象成四层：

```
知识源(输入) → 路由(判断) → 生成(执行) → 质量控制(检查)
```

**第一层：知识源。** 输入不是用户的自然语言描述，而是已有的知识源——代码、设计文档、DDL、接口规范、文章正文。不创造新格式，不维护额外的"架构描述文件"——知识应该放在它该放的地方，ai-viz 只是去读它。

**第二层：路由。** AI 根据知识源的类型和结构，自动判断该画什么图、用什么格式。这层是"智能"发挥的地方。

**第三层：生成。** 按照渲染规格和设计语言，生成具体的图表文件。这层是"执行"。

**第四层：质量控制。** 生成后不是结束。AI 对照检查清单自检，确保从"生成物"到"可交付物"之间的质量门禁通过。

对应到框架设计，是四层架构：

**Core 层：方法论内核**

定义"画什么图"的决策逻辑。包含路由规则（知识源→图种映射）、置信度分级策略（什么时候直接画、什么时候先确认）、渲染规格体系（每种图种的布局/配色/元素规范）、设计语言注入机制（`design-language.yaml`）、质量控制标准。

这一层跟输出格式无关，跟具体工具无关。

**Plugin 层：输出格式**

四个插件各管各的格式：
- **drawio** — 生成 XML，支持导出 PNG/SVG/PDF，适合对外正式场景
- **excalidraw** — 生成 JSON，手绘风，适合内部文档和白板讨论
- **mermaid** — 生成文本代码块，版本控制友好，适合 README
- **ian-illustrator** — 生成 PNG 配图，艺术手绘风，适合科普文章和博客

每个插件包含自己的指令文件、Schema 定义、质量自检清单。

**Adapter 层：工具适配**

同一套方法论 + 插件，需要输出到 10 个不同 AI 工具的原生指令目录。Claude Code 放 `.claude/skills/`，Cursor 放 `.cursor/rules/`，Copilot 放 `.github/copilot-instructions.md`。适配器负责把编译后的内容写到对的位置、对的格式。

**CLI 层：安装部署**

`npx ai-viz init` 一条命令，交互式向导引导选工具、选插件、选语言、生成配置。不需要手动复制文件。

为什么必须是插件架构？因为可视化格式在持续演进。明天出了一个新的图表工具，只需要加一个插件，不改框架。

## 第六章：智能路由——知识源到图种的自动映射

路由是整个方法论的核心差异化能力。而路由的本质，是 **知识源类型 → 图种** 的映射。

### 为什么路由的起点必须是知识源

传统模式是"你告诉 AI 画什么"。你得先判断该画架构图还是时序图，然后告诉它。但这里有两个隐藏的问题：

**第一，人经常判断不准。** 很多时候你只知道"这段设计文档需要一张图来辅助理解"，但不确定该画架构图还是数据流图。这个判断本身需要专业知识。

**第二，人判断的成本被忽略了。** 你得先理解知识源的结构特征，对比各种图种的适用场景，做出选择。这个"选图种"的过程有时比画图本身还花心思。

ai-viz 的思路是：你不需要判断。把知识源交给 AI，AI 从源的结构特征中自动推断该出什么图。DDL 里有表结构和外键关系——这只能是 ER 图。Swagger 里有请求链路——这天然是时序图。K8s YAML 里有服务和网络配置——这就是部署图。

**知识源的结构决定了图种，正如数据结构决定了算法。**

### 置信度分级：什么时候自动执行，什么时候问你

不是所有情况都能百分百确定图种。路由设计了一个置信度分级策略：

**第一优先级：用户明确指定。** 你说"画个时序图"，直接执行，没有废话。

**第二优先级：知识源强映射。** DDL 进来→ER 图，接口规范进来→时序图，K8s YAML 进来→部署图。这些映射关系确定性极高，AI 直接执行并附上理由。

**第三优先级：内容类型判断。** 技术工程内容→drawio/mermaid/excalidraw；科普文章/概念解释→ian-illustrator。这层确定性稍低，但仍足够高。

**第四优先级：多种可能。** 一份设计文档可能既适合架构图也适合数据流图——AI 给出推荐和理由，等你确认。

**第五优先级：高成本场景兜底。** 对外正式文档、需要额外工具链的格式（如 drawio 需要 CLI 导出 PNG），推荐方案但等确认再执行。

这是一个 **"确定性越高，自动化程度越高"** 的策略。减少交互摩擦，同时避免浪费生成成本。

### 路由还能主动出击

路由不只是被动等你"给知识源"。当 AI 在对话中识别到以下信号时，会主动建议生成图表：

- 描述涉及 3 个以上交互组件 → 建议架构图
- 出现时序性描述（先…再…然后…）→ 建议时序图
- 出现条件分支（如果…否则…）→ 建议流程图
- 出现分层/分级结构描述 → 建议分层架构图
- 出现对比/并列描述 → 建议对比矩阵图
- 内容面向非技术读者 → 建议 ian-illustrator 配图

不需要你想着"这里该画张图了"——AI 替你想。

## 第七章：质量控制——从"生成"到"可交付"

AI 能生成图了，但生成不等于可用。

如果你用过 AI 画图，一定遇到过这些情况：节点挤在一起看不清、连线交叉成蜘蛛网、重要信息被漏掉了、配色花得像调色盘爆炸。生成物和可交付物之间，有一道质量门禁。

传统方式的质量控制靠人眼——你看一眼，觉得"行"就行了。但这有两个问题：一是主观标准不稳定（今天觉得行，明天觉得不行），二是检查维度不全面（你可能漏看了连线标注缺失）。

ai-viz 的做法是：**把质量标准规则化，让 AI 自己对着清单检查。** 不靠感觉，靠规则。

### 五个核心检查维度

**1. 结构正确性——图的逻辑对不对**

这是最基础的维度：图表的逻辑结构是否准确反映了知识源。

检查内容：
- 知识源中的核心实体是否都在图中体现了（不能漏）
- 实体间的关系是否完整、准确（不能错）
- 图种选择是否匹配知识源类型（不能答非所问）
- 层次结构是否正确（该在上层的没掉到下层）

举个例子：你给了一份微服务设计文档，里面提到了网关、5 个业务服务、2 个数据库。生成的架构图里只画了 3 个服务——这就是结构正确性不达标。

**2. 布局合理性——看起来清不清楚**

结构对了但布局差，等于白画。没人愿意看一张节点挤成一团、连线交叉如麻的图。

检查内容：
- 无元素重叠
- 布局方向一致（不混合从上到下和从左到右）
- 元素间距均匀，留白充分
- 分层/分组逻辑清晰
- 复杂度与布局策略匹配（节点多时主动扩大间距）

复杂度分级策略：

| 复杂度 | 核心节点数 | 布局策略 |
|--------|-----------|---------|
| 简单 | ≤ 8 | 紧凑布局，单区域 |
| 中等 | 9-15 | 分组布局，2-3 个逻辑区域 |
| 复杂 | 16-20 | 分层布局，扩展路由走廊 |
| 超复杂 | > 20 | 拆分为多张图 |

**3. 信息完整性——该有的都有了吗**

一张图的价值在于它传达了信息。如果关键信息缺失，图就只是装饰。

检查内容：
- 所有连线有标注（说明传递的是什么：HTTP 请求？消息？数据？）
- 标签文字未截断，完整可读
- 节点命名一致（同一实体在全图使用相同名称）
- 关键约束/规则是否有标注（如"异步""可选"）

**4. 风格一致性——符合项目的视觉规范**

每个项目有自己的 `design-language.yaml` 定义视觉风格。生成的图要跟项目风格一致，而不是每次用随机配色。

检查内容：
- 配色符合设计语言定义的语义槽位（primary/secondary/accent/muted）
- 字体大小层次分明（标题 > 正文 > 注释）
- 同类型元素使用相同形状和尺寸
- 间距和对齐遵循网格

**5. 可交付性——格式正确、导出可用**

图画得再好，导出的文件打不开等于零。

检查内容：
- 格式校验通过（Excalidraw 的 JSON 可解析、Draw.io 的 XML 标签正确闭合、Mermaid 语法可渲染）
- 元素 ID 唯一（不出现重复 ID 导致渲染异常）
- 连线端点引用正确（不出现"指向不存在的元素"）
- 特殊字符正确转义

### 每种格式有额外的专项检查

通用维度之外，不同输出格式有自己的"坑"：

**Excalidraw 专项：**
- JSON 合法性（能被解析器正确解析）
- 箭头的 startBinding/endBinding 指向正确元素
- groupIds 关联正确，无孤立分组引用
- 坐标合理，元素不超出画布范围

**Draw.io 专项：**
- XML 标签正确嵌套闭合
- mxCell 的 source/target 引用的 id 存在
- style 属性包含必要的视觉参数
- parent 属性指向正确的容器

**Mermaid 专项：**
- 语法可被 Mermaid 渲染器解析
- 节点 ID 不含导致解析失败的特殊字符
- subgraph 正确闭合
- 含特殊字符的标签用引号包裹

### AI 自检 vs 人工检查：本质区别

传统质量控制依赖人工 review。人看一眼："还行，节点位置可以再调调。" 这种方式的问题是：

1. **标准不稳定**——同一个人不同时间判断不同
2. **维度不全面**——人容易漏掉格式层面的问题（比如 ID 重复）
3. **不可重复**——换一个 reviewer 标准完全不同
4. **效率低**——每张图都得人看一遍

ai-viz 的做法是把检查逻辑写成规则清单，AI 生成图之后自动逐项检查。发现问题当场修复，不用等人来 review。

这不是说人的判断没有价值——复杂场景下的"这张图是否真正表达了设计意图"仍然需要人来把关。但 80% 的质量问题（格式错误、元素遗漏、布局混乱）可以被规则化的自检捕获。

### Create 模式和 Edit 模式的不同检查策略

**Create 模式（从零生成）：** 全量检查。所有维度跑一遍，确保首次输出就达标。

**Edit 模式（增量编辑）：** 聚焦检查。只检查修改涉及的区域——新增的节点是否有标注、新增的连线端点是否正确、修改部分的布局是否与周围元素冲突。同时确保"最小变更原则"——未涉及的节点保持原有位置和样式。

Edit 模式额外的约束：
- 每次编辑保留已有布局，只修改涉及的部分
- 保持修改可追溯（告知用户改了什么）
- 同一图表最多 5 轮迭代，超过建议拆分或重新梳理需求

### 什么时候应该放弃修改，重新生成

有些情况下，在旧图上修修补补不如重来：
- 需求变更超过 50%
- 图种需要变更（比如从流程图改为时序图）
- 布局结构根本性变化（比如从 3 层改为 5 层）
- 迭代超过 5 轮仍未收敛

这时候 AI 会主动建议切换回 Create 模式重新生成。

## 第八章：10 个工具，一次安装

方法论设计好了，面临一个现实问题：每个 AI 工具的指令格式和存放位置都不一样。

Claude Code 要求技能放在 `.claude/skills/` 目录下，每个技能一个文件夹带 `SKILL.md`。Cursor 要求规则放在 `.cursor/rules/` 下，用 `.mdc` 格式。Copilot 只读项目根目录的 `.github/copilot-instructions.md`。Codex /ˈkoʊ.dɛks/ 读 `codex.md`。

如果让用户自己搞，安装过程就变成了噩梦。

解决方案是 **适配器模式**。每个 AI 工具对应一个适配器，适配器知道目标工具的目录结构、文件格式、命名约定。编译器把核心方法论 + 选中的插件 + 设计语言配置组装好，适配器负责写到正确位置。

目前支持 10 个工具：

| 工具 | 输出位置 |
|------|----------|
| Claude Code | `.claude/skills/ai-viz/` |
| Cursor | `.cursor/rules/` |
| Windsurf | `.windsurf/rules/` |
| OpenCode | `.opencode/skills/ai-viz/` |
| GitHub Copilot | `.github/copilot-instructions.md` |
| Codex | `codex.md` |
| Qoder /ˈkoʊ.dər/ | `skills/ai-viz/` |
| Trae /treɪ/ | `.trae/rules/` |
| CodeBuddy | `.codebuddy/rules/` |
| Aider /ˈeɪ.dər/ | `.ai-viz/` |

用户体验就一条命令：

```bash
npx ai-viz init
```

交互式向导会问你用什么工具、要什么插件、什么语言。30 秒完事。之后你在项目里跟 AI 说"画个架构图"，它已经知道该怎么做了——因为方法论已经在它的指令里。

## 第九章：理念——人负责思考，AI 负责表达

回到起点。

画图是表达行为，不是创作行为。既然是表达，就应该被标准化、被自动化。

这个想法推广到更大范围也成立：

- 写技术文档的人，时间应该花在"把技术讲清楚"上，不是"选什么工具画流程图"上
- 写科普文章的人，时间应该花在"找到好的隐喻"上，不是"手动做配图"上
- 做教程的人，时间应该花在"设计教学路径"上，不是"调图表配色"上

**人负责思考，AI 负责表达。** 这是 ai-viz 的核心设计哲学。

而这一切的起点，是对"知识源"的重新认识——图的来源不是一段描述文字，是已经存在于各种载体中的知识。ai-viz 做的事情，是链接到这些知识源，理解它们的结构，选择最合适的可视化方式，按规格渲染，经过质量门禁验证，输出可交付的图表。

从一次画图时的灵感出发，经过三个开源 Skill 的发现和改造，抽象出"知识源→路由→生成→质量控制"的通用范式，最终做成一个四层插件架构、覆盖 10 个工具的开源框架。

路不长，但每一步都是从实践中长出来的。
