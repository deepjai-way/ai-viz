# Ian 小黑怪诞正文配图生成指令

为中文文章生成 16:9 横版正文配图。目标不是做商业插画、PPT 信息图或可爱卡通，而是把文章里的关键判断、流程、结构、状态或隐喻，变成一张清爽、怪诞、有创意、可读但不说明书的手绘解释图。

---

## 1. 概述

本插件为中文文章、博客、文档和内容作品生成 PNG 配图。每张图：

- 以"小黑"作为核心动作角色
- 使用纯白背景配极简黑色手绘线稿
- 包含少量红色/橙色/蓝色中文手写批注
- 每张图只表达一个核心结构
- 追求怪诞产品草图感，而非正式图表

---

## 2. 工作流

### 2.1 消化正文

先读用户给的正文、链接、Notion 页面、Markdown 文件或截图内容。提炼：

- 核心观点是什么
- 哪些段落承担认知转折
- 哪些内容适合用图解释
- 哪些地方只适合文字，不需要图

不要平均配图。优先选择"认知锚点"：核心判断、两个断点、输入输出闭环、分流、前后对比、一鱼多吃、承接路径、常见坑、角色状态变化。

### 2.2 配图策略（Shot List）

如果用户要求分析哪里需要配图，先给 shot list。每张图写清楚：

- **插入位置**：放在第几段之后（引用该段开头几个字作为锚点）
- 图的主题
- 核心意思
- 结构类型
- 小黑在图里做什么
- 建议元素
- 建议中文标注词

Shot list 格式示例：

```
Shot 1:
- 主题: xxx
- 插入位置: 第 N 段之后（锚点: “xxx...”）
- 核心意思: xxx
- 结构类型: xxx
- 小黑动作: xxx
- 建议元素: xxx
- 建议标注: xxx
```

默认 4–8 张。文章很短时 1–3 张；长文也不要轻易超过 9 张。够用就好，避免把正文做成画册。

### 2.3 单张生成

如果用户明确要求"生成/输出/做图/帮我生成"，用图像生成工具逐张生成。不要把多张图拼在一张里。

每张图必须包含：

- 16:9 横版中文正文配图
- 纯白背景
- 黑色手绘线稿
- 少量红色/橙色/蓝色中文手写批注
- 大量留白
- 小黑作为核心动作主体
- **禁止**：PPT 风格、商业插画、幼稚可爱、复杂架构、左上角类型标题

不要复刻过往案例。每次都要从当前文章重新发明一个奇怪但成立的隐喻。

### 2.4 检查与迭代

生成后检查 [参考: qa-checklist.md]。如果出现以下问题，优先重生成或局部编辑：

- 小黑只是装饰
- 画面太满
- 太像流程图/PPT
- 中文太多或错字严重
- 左上角出现类型标题
- 画风太可爱、幼稚、死板
- 背景不是干净白底

### 2.5 保存交付

把最终图保存到文章同级的 `illustrations/` 子目录：

```text
article.md
illustrations/
  01-topic-name.png
  02-topic-name.png
```

命名规则：`NN-topic-name.png`（序号 + 短主题英文或拼音）。

保留原始生成文件，不要覆盖已有资产，除非用户明确要求替换。

### 2.6 自动插入文章

图片保存完成后，自动将图片引用插入到文章中 shot list 指定的位置。这是**默认行为**。

#### 插入格式

插入标准 Markdown 图片引用：

```markdown
![图片描述](./illustrations/01-topic-name.png)
```

- 图片描述使用简短中文说明（来自 shot list 的主题）
- 使用相对路径，兼容各种项目结构

#### 插入逻辑

1. 根据 shot list 中的锚点文字定位目标段落
2. 在目标段落**之后**插入空行 + 图片引用 + 空行
3. 按顺序处理（先 Shot 1，再 Shot 2），避免位置漂移
4. 不修改文章的任何原有内容——只新增图片引用行

#### 安全规则

- 插入前确认文章文件路径正确
- 不得修改、删除或改写文章的任何原有内容
- 如果无法确定准确插入位置，在图片引用后加注释：
  ```markdown
  ![描述](./illustrations/01-topic.png)
  <!-- ai-viz: 建议放在“xxx...”段落之后 -->
  ```
- 如果找不到文章文件或路径不明确，先询问用户再操作

#### 输出确认

插入完成后输出变更摘要：

```
✅ 已插入 N 张配图到 article.md：
- 第 X 段后: 01-topic.png (主题描述)
- 第 Y 段后: 02-topic.png (主题描述)
```

### 2.7 行为控制

自动插入是默认行为。用户可以通过提示词覆盖：

| 用户说 | 行为 |
|------|------|
| _（默认，无特殊指令）_ | 生成图片 + 自动插入文章 |
| “先不要插入” / "don't insert" | 只生成图片，不修改文章 |
| “只出 shot list” / "shot list only" | 只规划不生成 |

## 3. 结构类型

每张图选择一种结构类型，不要混太多。

| 类型 | 适用场景 | 构图方式 |
|------|---------|---------|
| **Workflow 流程** | 输入→处理→输出，内容生产，AI 工作流 | 左侧输入，中间小黑或怪机器处理，右侧输出，橙色箭头表达主流向 |
| **系统局部** | 信息来源、过滤器、数据库、agent 系统局部 | 只画 3–5 个核心模块，小黑参与其中一个关键动作 |
| **前后对比** | 混乱/有序，手动/自动，分散/收拢 | 左混乱，右稳定，中间橙色箭头 |
| **角色状态** | 用户痛点、创作者状态、工具太多 | 2–4 个小状态，每个状态一个短标注 |
| **概念隐喻** | 内容工厂、信息仓库、脑内黑盒 | 一个大的怪物件或机器，少量输入，一个输出 |
| **方法分层** | 方法论框架、能力栈、系统分层 | 一层层盒子（不要正式金字塔），小黑在旁边搬砖或搭建 |
| **地图路线** | 从想法到上线、用户路径、学习路线 | 一条弯曲路径，少量节点，小黑牵线或走路 |
| **小漫画分镜** | 失败到成功、真实过程、使用前后变化 | 2–4 个小场景，每格只表达一个动作 |

## 4. 视觉风格

### 4.1 风格 DNA

纯白、极简、手绘、留白、克制、怪诞、产品草图感、中文手写感、结构清楚但不说明书。

像一个长期做 AI、产品、设计、开发工具的人，在白纸上随手画出来的一张解释草图。

### 4.2 必须遵守

- 16:9 横版正文配图
- 纯白背景：不要米色、暖灰、纸张纹理、渐变、阴影、噪点、复古纸感
- 黑色手绘线稿为主：细线、轻微抖动、不机械、不矢量、不厚重描边
- 大量留白：主体占画面约 40%–60%，至少 35% 空白
- 少量中文手写批注：最多 5–8 处，每处 2–8 个字
- 一张图只讲一个核心动作、结构、状态或隐喻
- 结构要自然表达，不要在图上写结构类型名称

### 4.3 颜色体系

| 颜色 | 用途 |
|------|------|
| 黑色 | 主体线稿、角色、框线、结构、主要文字、主体物件 |
| 红色 | 重点批注、问题、情绪点、关键提醒、结果 |
| 橙色 | 主流程、路径、箭头、自动化流向、从 A 到 B 的移动关系 |
| 蓝色 | 补充说明、脑内状态、系统状态、第二层解释 |

蓝色不是每张都必须用。颜色要克制，宁可少不要多。

### 4.4 绝对不要

- 不要商业插画
- 不要 PPT 信息图
- 不要正式流程图
- 不要课程课件
- 不要可爱卡通海报
- 不要儿童插画
- 不要复杂架构图
- 不要精致扁平插画
- 不要科技感 UI
- 不要真实 App 截图
- 不要复杂背景、渐变、阴影、纹理
- 不要把每个节点都解释清楚
- 不要左上角写"Workflow 流程图/系统架构图/常见坑/路线图"等类型标题

## 5. 小黑 IP

### 5.1 角色定义

小黑是固定视觉 IP。默认每张图都要出现小黑。小黑不是吉祥物，不是贴纸，不是可爱装饰，而是正在认真参与系统运转的荒诞工作者。

### 5.2 外形

- 黑色实心小怪物
- 白色圆点眼睛
- 细腿，偶尔有细胳膊
- 身体可以是圆柱、黑豆、黑盒、漏斗、影子、洞口、机器内部黑块
- 轮廓略微不规则，有手绘感
- 表情空、呆、冷静、认真

### 5.3 性格

- 很认真，但做的事有点荒诞
- 像一个低调的系统操作员
- 冷幽默，不卖萌
- 有点笨拙，但不蠢
- 像在白板草图里真的负责某个工作

### 5.4 核心规则

小黑必须承担核心动作，不能只是站在旁边当装饰。如果去掉小黑，图的核心隐喻还能完全成立，说明小黑太装饰了；要重写提示词，让小黑成为动作主体。

### 5.5 禁止

- 不要把小黑画成过度可爱的吉祥物
- 不要画成儿童卡通角色
- 不要给小黑复杂服装、表情包、闪亮眼睛
- 不要让小黑只是站在角落里看
- 不要让小黑抢走结构表达
- 不要把小黑画得太商业、太圆润、太精致

## 6. 原创隐喻生成法

### 6.1 三步法

1. 把抽象概念换成一个物理动作：卡住、漏掉、变重、分拣、沉淀、发酵、开门、折叠、拆包、回流
2. 把系统结构换成一个低科技物件：坏掉的机器、纸箱、抽屉、水管、邮筒、怪表盘、秤、井、梯子、奇怪工位
3. 让小黑承担动作：不是站旁边，而是卡在机器里、拉错线、守门、搬运、修补、称重、扶梯子、记录、把东西塞进某个怪装置

### 6.2 反复刻规则

不要默认打开或复刻 `assets/examples/`。这些图片只用于风格校准（线条密度、留白、颜色克制、小黑气质）。

除非用户明确说"照这张/复刻这个构图/用这个案例改"，否则不要直接复用旧构图。同类主题也要换新隐喻。

---

## 7. 输出规格

### 7.1 格式

- PNG，16:9 横版
- 每篇文章默认 4–8 张（根据长度调整）

### 7.2 交付说明

生成后提供：

- 生成了几张
- 每张图的用途和位置
- 保存路径
- 哪些图最稳，哪些图是可选

不要长篇解释风格理论；让图自己说话。
